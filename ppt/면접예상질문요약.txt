SKU (Stock Keeping Unit) 는
 재고를 관리하기 위해 상품 하나하나에 부여하는 고유 코드입니다.

React는 화면 담당
Spring Boot는 데이터 + 로직 담당
React가 axios / fetch로 API 요청
Spring Boot가 JSON으로 응답

“React 개발 서버(localhost:3000)에서
Spring Boot API(localhost:8080)를 호출해서
회원가입, 로그인, 상품 조회 기능을 연동했습니다.”

이 프로젝트의 기술적 강점 3가지
백엔드 구조 설계

Controller / Service / Repository 계층 분리로
유지보수성과 확장성을 고려한 구조
>>>
-Controller
 요청을 받는 입구 (HTTP 요청/응답)
-Service
 실제 비즈니스 로직 처리
-Repository
 DB와 직접 통신

② 인증·권한 처리
JWT 기반 인증 + ROLE_ADMIN 권한 체크로
실무에서 사용하는 보안 구조 적용

③ 실무형 설정
공통 예외 처리와 .gitignore 설정으로
실제 서비스 환경을 고려한 프로젝트



JWT 기반 인증·인가
JWT는 로그인 후 서버가 발급하는 토큰으로,
사용자의 인증 정보와 권한을 담아 요청마다 전달하는 방식입니다.
로그인 성공 시 JWT 토큰을 발급하고,
이후 요청마다 토큰으로 사용자를 인증했습니다.

세션이 아니라 JWT를 쓴 이유는
토큰 자체는 암호화되지 않기 때문에
REST API 구조에 맞는 무상태(stateless) 인증을 사용하기 위해서입니다.
서버가 로그인 상태를 저장하지 않아 확장성과 성능에 유리합니다.
프론트엔드에서 로그인 후 토큰을 저장하고,
이후 API 요청 시 Authorization 헤더에 담아 전송했습니다.
만료 시간 설정과 시크릿 키 관리가 중요합니다.
실제 프로젝트에서도 만료 시간과 시크릿 키를 설정했습니다.

흐름
로그인 성공
서버가 JWT 토큰 발급
React가 토큰을 저장
API 요청 시 Authorization 헤더에 토큰 전송
서버가 토큰 검증
(로그인 → JWT 발급 → 프론트 저장 →
API 요청 시 토큰 전송 → 서버에서 토큰 검증 → 사용자 인증)

핵심 포인트
세션을 쓰지 않는 방식
서버가 로그인 상태를 저장하지 않음 (무상태)
REST API에 적합한 인증 방식

관리자 권한 체크 (ROLE_ADMIN)
JWT 안에 사용자 권한을 담아 상품 등록이나 관리자 전용 API는
ROLE_ADMIN 권한이 있는 사용자만 접근 가능하도록 설정했습니다
>>>>
-일반 사용자 → 상품 조회만 가능
-관리자 → 상품 등록 / 수정 / 삭제 가능

예외 처리 (@ControllerAdvice)
에러를 한 곳에서 관리해서
코드 중복을 줄이고 응답 형식을 통일하기 위해서입니다.
>>인증 실패나 잘못된 요청 같은 예외를
공통 처리해서 프론트엔드에서 에러 대응이 쉽게 하기 위해서입니다.
-왜 필요한가?
에러를 Controller마다 처리하면 코드 중복 + 관리 어려움이 있습니다.

@ControllerAdvice로
인증 실패/데이터 없음/잘못된 요청/등을 한 곳에서 처리

.gitignore 설정-보안
민감한 정보가 깃허브에 올라가지 않도록
.gitignore를 설정했습니다.
설정 파일과 보안 정보를 깃허브에 올리지 않기 위해
.gitignore를 적용했습니다.
실무에서는 보안 사고의 90%가 설정 파일 노출에서 시작되기때문입니다

숨긴정보
application.yml / properties
DB 비밀번호
JWT 시크릿 키
로컬 설정 파일

@ManyToOne(fetch = FetchType.LAZY) 를 사용한 이유는
ManyToOne은
주문 여러 개가 회원 한 명을 참조하는 것처럼
N : 1 관계를 표현하는 어노테이션입니다
연관 엔티티를 즉시 조회하지 않고, 필요할 때 조회하는 전략으로
“ManyToOne 관계에서 성능 문제를 방지하기 위해 
LAZY 로딩으로 변경하여
실제로 연관 객체를 사용할 때만 조회되도록 했습니다
